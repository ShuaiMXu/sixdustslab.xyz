Claude 那 422 行被泄露的设计哲学

AI 时代的设计护城河,比 Figma 想得更深一层。

"比 Claude Design 这个产品更值钱的,是它把'如何做好设计'翻译成了 400 多行可执行规则的能力。"

— 一个写设计 Skill 的人

最近一个 GitHub 仓库 CL4R1T4S 泄露了 Anthropic Claude Design 的系统提示词,标题党们说有 422 行——具体行数我没数,但读完只想说一句:

人们都在看产品,我看的是这份 prompt 本身。

因为它揭示了一件比 Claude Design 火不火更重要的事:AI 时代,设计的护城河长什么样。

SECTION 01 · IGNITION

引爆点不在 Claude Design 本身

这事的时间线是这样的——

Claude Design 上线那一周,Figma 股价跌了一波,设计圈炸了:终于有一个能"看懂设计哲学"的 AI 工具,而不是再多一个"复刻 UI 的玩具"。

然后 GitHub 上有人把它的系统提示词扒了出来,完整贴在一个 Markdown 文件里。

看完那份 prompt 的人都意识到:

Claude Design 这个产品迟早会被对手抄超,但它把"美学"翻译成"可执行规则"的能力——这才是真正的护城河。

泄露版里有一段话尤其值得贴出来——这是整份 prompt 的灵魂:

"You tend to converge toward generic, 'on distribution' outputs. In frontend design, this creates what users call the 'AI slop' aesthetic. Avoid this: make creative, distinctive frontends that surprise and delight."

翻译过来:

"你倾向于塌缩到平庸的中位数。在前端设计里,这种现象被用户叫做 AI slop(AI 泔水)。避免这个,做出有创意、有辨识度、能让人惊喜的前端。"

这段话最反直觉的不是"避免平庸"——所有设计文档都这么说。而是 Anthropic 承认了大模型的统计本性:在没有约束的情况下,模型默认会输出训练数据的高频中位数。

用"slop"(泔水)这个相当不客气的词命名这个失败模式,然后把"对抗 slop"明明白白写进操作守则——这就是这份 prompt 的写作姿态。

SECTION 02 · UNPACKED

拆开 422 行 · 六条核心哲学

整份 prompt 我读了两遍,提炼出 6 条核心哲学。每一条都是"模型默认会怎么烂 → 我们用什么具体规则把它扳回来"——一份完整的反塌缩工具箱。

01
承认 AI 是统计塌缩,然后和它对抗
整份 prompt 没有假装 AI 是创意天才,反而承认 "defaults to safe choices unless explicitly encouraged otherwise"。结构是反的——每章都是"它默认会塌缩到哪 → 用什么规则扳回来"。

启示:写 Skill / Agent 时,第一性问题是"它默认会怎么平庸",不是"它能做什么"。
02
把"美学"拆成四个调音旋钮
不是一句"做得好看点",而是字体 / 颜色 / 动效 / 背景四个独立维度,每个都给"避免清单 + 推荐清单 + 对比原则"。

字体:"Never use: Inter, Roboto, Open Sans, Lato, default system fonts"。
颜色:"Avoid clichéd color schemes (particularly purple gradients on white backgrounds)"。

启示:审美的 SOP 化是可能的——前提是把"风格"拆成可机读的维度。
03
高对比 / 极端 / 大跨度,优于均匀
"Use extremes: 100/200 weight vs 800/900, not 400 vs 600. Size jumps of 3x+, not 1.5x."

字重要么 100 要么 900,别用 400 vs 600。字号跳 3 倍,别跳 1.5 倍。颜色要主导色 + 锐利点缀,别均匀分布。

启示:平庸 = 中位数。差异 = 极端。
04
设计上下文优先于设计灵感
模型被要求先吃代码 / 读现存 design system / 找不到组件就主动问用户要,而不是凭空捏造。"原创"建立在"先理解既有约束"之上。

官方原话: "Claude is better at recreating or editing interfaces based on code, rather than screenshots."

启示:好的 AI 设计师 ≠ 灵感生成器。好的 AI 设计师 = 上下文吸收器。
05
自检是默认动作,不是可选项
Claude Design 内置了一个 fork_verifier_agent:模型完成 → 派一个后台子 agent 截图 + 探针检查布局 → 报告。

这不是"如果有时间检查一下",而是每次任务的强制环节。

启示:Agent 的可靠性来自架构层强制的自检环路,不是来自模型本身能力。
06
伦理是操作步骤,不是抽象原则
被要求复刻其他公司 UI 时,Claude 会拒绝——除非用户的邮箱域名能证明他来自那家公司。绕开方式: 理解用户真正想构建什么,然后帮他做尊重 IP 的原创设计。

启示:伦理可以是用邮箱域名做硬性 gate + 明确转化路径这种具体动作,不一定是抽象口号。
SECTION 03 · ISOMORPHISM

巧合的是,我们也走在同一条路上

读完那 6 条哲学,我有种奇妙的"似曾相识"感。

因为我所在的设计团队最近正在做一个项目——把某 App 的设计 wiki 升级到 AI 时代真正能用的活系统。背景是这样的:

这个 App 是多业务线共生的超级平台,不同业务线各自做自己的内容卡片,导致用户跨业务线时心智断裂——同一种"内容物种",在 A 场景看到和 B 场景看到长得不一样,体验割裂。

我们花了一段时间做了一个具体场景的样本——把"内容卡片"从混乱状态梳理成有结构的设计语言。结果发现:

我们独立得到的方法论,和 Claude Design 那 422 行系统提示词里的哲学,几乎是同构的。

对照表如下:

CLAUDE DESIGN SYS PROMPT 我们的实践
9 个 aesthetic family
(Editorial / Terminal / Cinematic Dark / Glass-Soft Futurism ...)		 5 大业务家族
(同样的"家族识别"哲学,只是按业务功能纵切)
每个 family 的"底部不变量"锚定身份 家族 DNA 由底部不变量锚定
(我们的 L1-5 核心规约)
反"AI slop"——避免塌缩到平均值 观察不是判决
抗的也是设计师塌缩到中位数
CSS 变量绑死品牌色
不发明新色		 原则 8:动态数据不作家族识别硬条件
不要凭空发明判定信号
fork_verifier_agent 自检环路 PE 模板里的自检清单
子 agent 独立评分防自欺
Skill 是 ~400 token 可挂载上下文 把样本做成开源 Skill
跨 agent 可复用

这不是巧合。

这是 AI 时代企业级设计系统的必然形态——把设计师直觉解码成 family + 规则 + 自检流程。任何一个团队,只要认真做"AI 时代的设计治理",最后都会得出几乎一样的架构。

你可以从 Claude Design 的 prompt 倒推,我们从一个具体场景的失败样本正推,殊途同归。

SECTION 04 · ACTION

中文团队如何写自己的 422 行

读完 Claude Design 的 sys prompt + 对照我们自己的实践,有 5 个具体动作可以落地——这是中文团队写自己 422 行的清单:

A1
写一份"反塌缩清单"
列出"AI 在你这个行业最常犯的平庸错"——比如紫色渐变、Inter 字体、卡片圆角全 8px、shadow 叠 3 层。把这些显式写成"绝对不要做"清单,放进每次任务的执行约束里。
A2
设计上下文先行
Skill 在分析新输入前,强制先扫一遍同业务线已存在的家族基线。这相当于 Claude 的"先吃 code 而非 screenshot"。
A3
双阶段验证
产出报告 + 派一个对抗 agent 用挑刺视角再扫一遍——"是否硬塞了不存在的差异?是否漏掉了真正反常的样式?"。同一个反自欺哲学,可以拓展到任何专项 Skill。
A4
Tweaks 协议对应物
Claude Design 让用户在原型里直接圈选+微调,而不是来回打字。中文团队的对应物可以是:让设计师在卡片图上"圈选疑问区域→自动展开 Skill 解释",而不是看完静态报告再翻规范。
A5
把"伦理"翻译成操作步骤
跨业务线借鉴的 gate 流程——不是禁止借鉴,而是规定借鉴需要走"原型理解 → 抽象 token → 重新具象"。把"尊重"做成具体动作,不留口号空间。
SECTION 05 · MOAT

AI 时代设计的真正护城河

Figma 这一波不是"做得不好"。是它的护城河被重新定义了。

原来 Figma 的护城河是用户习惯 + 协作网络。当自然语言成为交互界面时,这两层都可以被绕过——用户可以直接告诉 AI"做一个登录页面",不需要先学怎么用 Figma。

所以问题变成了:AI 时代的设计护城河,到底在哪一层?

第 ① 层
应用层
(脆弱)
设计工具、生成模型壳子。一旦更强的工具出现,直接被替换。
第 ② 层
模型层
(巨头垄断)
大模型本身。Anthropic / OpenAI / Google 的战场,一般团队进不去。
第 ③ 层
能源算力
(NVIDIA)
GPU、电力、数据中心。设计团队跟这层完全无关。

设计团队既上不了第 ② 第 ③ 层,又不能困在被随时替换的第 ① 层。那真正的位置在哪?

读完 Claude Design 的 422 行后,我想清楚了一件事:

AI 时代设计的真正护城河,
不在生成能力,
在反塌缩规则的工程化。

— THE REAL MOAT

大模型公司战的是"模型有多强"。设计团队战的应该是"我能不能把我的设计判断,翻译成模型可执行的规则"。

Claude Design 那 422 行就是这种能力的范本——Anthropic 在用一个产品 demo,展示他们对"设计哲学如何工程化"的理解。这种理解本身,比那个产品值钱得多。

SECTION 06 · UPGRADE

给设计师的两个建议

这阵子很多人在讨论"AI 会不会替代设计师"。这是个错误的提问。

正确的问题是:设计师有没有把自己的判断升级为可被 AI 执行的资产?

给同行两个具体动作:

建议 ①
把你的设计判断写成规则
从最熟悉的一个组件开始——把它的"什么时候用、什么时候不用、关键不变量是什么、变体边界在哪"明确写成 100 行(不需要 422)。

不写下来 = 那条规则不存在,也就不能被复用、不能被 AI 消费、不能成为团队资产。
建议 ②
找一个具体场景跑通治理样本
不要从"建一套设计系统"开始,从"治理一个具体组件"开始。把 family 识别、规则定义、自检流程跑通一次。

小样跑通后,才有资格谈推广。否则你做的就是 PPT,不是基建。

好的 AI 设计师,不是灵感生成器。
是上下文吸收器 + 自检环路。

Claude 那 422 行系统提示词揭示的,就是这件事——把"如何做好设计"翻译成 AI 可执行的规则,本身就是 AI 时代最深的护城河。
